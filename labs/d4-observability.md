# Lab d4: Observability — metrics, logs, traces, dashboards, alerting

Stand up the open-source observability triad, then prove each signal fires: a PromQL alert that reaches Alertmanager, a LogQL query that finds a security event in Loki, and a trace that lands in Tempo — all correlated in one Grafana pane.

**Objectives covered**

|---|---|

## Challenge

Stand up the triad and reach five independent observables, then correlate them in one pane:

1. **Metrics.** A security-relevant PromQL query returns live series from the cluster, and Prometheus **Status → Targets** shows the `ServiceMonitor`-discovered endpoints `UP`.
2. **Logs.** A LogQL query surfaces `Failed password` lines from a noisy demo app in Loki, and a second query turns that log stream into a rate that visibly climbs.
3. **Traces.** A multi-span trace (frontend → backend, generated by real traffic) is searchable in Tempo with correct parent/child spans.
4. **Dashboards.** Killing the Grafana pod loses no data or dashboards — proving Grafana is stateless visualization over the other three backends.
5. **Alerting.** `AuthFailureSpike` transitions `Pending`→`Firing` in Prometheus **and** shows up grouped/routed in Alertmanager; a silence mutes delivery without stopping the rule — the fired-alert-reaching-Alertmanager is the proof.

No PromQL, LogQL, or alert-rule YAML below — write the queries and the `PrometheusRule` yourself in the guided section, then check your work against the reference solution.

## Build it (guided)

### Part A — Metrics with Prometheus (`obs-metrics`)
1. Confirm the stack: `kubectl get pods -n oss500-monitoring` — Prometheus, Alertmanager, Grafana, Loki, Tempo, and the node/kube-state exporters are Running.
2. Port-forward Prometheus: `kubectl -n oss500-monitoring port-forward svc/oss500-kube-prometheus-stack-prometheus 9090:9090`. Open `http://localhost:9090`.
3. **Write three security-relevant PromQL queries** in the expression browser — the goal is given, the syntax is yours:
   - **Pod sprawl per namespace.** `kube_pod_info` carries a `namespace` label on every pod series; you want a count of pods, grouped by that label. Which aggregation operator collapses a label into groups?
   - **Restarting containers.** `kube_pod_container_status_restarts_total` is a counter that only goes up. Which comparison finds every container that has restarted at least once?
   - **API-server error rate.** `apiserver_request_total` carries a `code` label. You want the per-second rate of 5xx responses over a 5-minute window — combine a regex match on `code` with the rate function.

   Your turn: write all three and confirm each returns a non-empty series.
4. Prove target discovery is label-driven: **Status → Targets** shows the `ServiceMonitor`-selected endpoints. Look at [`prometheus-values.yaml`](../lab-infra/observability/prometheus-values.yaml) — no hand-edited `prometheus.yml`; scrape config is CRDs.

### Part B — Logs with Loki (`obs-logs`)
5. Grafana is at `http://localhost:8080` via ingress (or `kubectl -n oss500-monitoring port-forward svc/oss500-grafana 3000:80`). Log in with the admin creds from `grafana-admin.env`.
6. **Explore → Loki** data source. Generate a security-flavored log: deploy the noisy demo app that logs auth failures — `kubectl apply -f ../lab-infra/observability/demo-authlog.yaml -n oss500-apps`.
7. **Write a LogQL query that finds the brute-force lines.** LogQL's log-stream selector looks like a Prometheus label selector — `{namespace="...", app="..."}` — narrow it to the demo app, then pipe (`|=`) into a line-contains filter for the string it logs on every failed attempt.

   Your turn: write the query and confirm the `Failed password` lines appear.
8. **Turn that log stream into a rate** — the basis for a log alert. LogQL supports `rate(...)` over a range the same way PromQL does, wrapped around a log-stream selector plus filter, then summed.

   Your turn: wrap your step-7 selector in a `sum(rate(...))` over a short window (e.g. `1m`) and watch the rate climb — you'll need this same shape of reasoning (rate crossing a threshold) in Part E.

### Part C — Traces with OpenTelemetry + Tempo (`obs-traces`)
9. Deploy the instrumented demo that emits OTLP to the Collector: `kubectl apply -f ../lab-infra/observability/demo-traced-app.yaml -n oss500-apps`. It calls a downstream service so traces have child spans.
10. Generate traffic: `kubectl -n oss500-apps port-forward svc/traced-frontend 8000:8000` then `for i in $(seq 1 20); do curl -s localhost:8000/api >/dev/null; done`.
11. Grafana **Explore → Tempo → Search**: find recent traces, open one, and read the span tree (frontend → backend). The `traceparent` header propagation is what stitched them.
12. Inspect the pipeline in [`otel-collector.yaml`](../lab-infra/observability/otel-collector.yaml): receivers (OTLP) → processors (batch) → exporters (Tempo, plus Prometheus/Loki) — one Collector, three backends.

### Part D — Dashboards & correlation in Grafana (`obs-dashboards`)
13. Import the provisioned **OSS-500 posture dashboard** (already loaded via the sidecar; find it under Dashboards). Panels: API-server 5xx rate, pods-running-as-root count, denied/failed-auth log rate, trace latency.
14. Prove cross-signal drill-down: from the failed-auth panel (Loki), click through to the raw log lines; from a latency spike, jump to the Tempo trace. This metrics→logs→traces pivot is the "single pane" investigation.
15. Confirm Grafana stores nothing itself: `kubectl -n oss500-monitoring delete pod -l app.kubernetes.io/name=grafana`, wait for it to restart, reopen — the dashboards (provisioned JSON) and all data (in Prometheus/Loki/Tempo) survive.

### Part E — Alerting rules + routing (`obs-alerting`)
16. **Design a `PrometheusRule`** that fires `AuthFailureSpike` when the failed-auth rate crosses a threshold. Don't just read the finished rule — decide the pieces yourself first:
    - Which metric/expression captures "the rate is spiking" (you already reasoned through this shape in Part B)?
    - What `for:` duration prevents the alert from flapping on a single transient blip, without also alerting too slowly?
    - What labels (severity? team?) would the security team's Alertmanager route need to pick this alert out from routine noise?

    Sketch your `PrometheusRule` (package it under a `group` in the `monitoring.coreos.com/v1` `PrometheusRule` CRD, in the monitoring namespace, with the release label the kube-prometheus-stack `ruleSelector` expects) before checking it against the reference.
17. Trigger it: scale the authlog app so failures spike — `kubectl scale deploy/authlog -n oss500-apps --replicas=3`. Watch **Prometheus → Alerts**: `AuthFailureSpike` goes `Pending` (during `for:`) then `Firing`.
18. Watch routing: port-forward Alertmanager (`kubectl -n oss500-monitoring port-forward svc/oss500-kube-prometheus-stack-alertmanager 9093:9093`), open `http://localhost:9093` — the alert is grouped by `namespace`/`alertname` and routed to the `security` receiver (webhook) per [`alertmanager-config.yaml`](../lab-infra/observability/alertmanager-config.yaml). Check whether the labels you picked in step 16 actually match that route's matchers.
19. Prove routing ≠ condition: add a silence in the Alertmanager UI for `alertname=AuthFailureSpike`; the rule still *fires* in Prometheus but delivery is muted — the two layers are independent.

## Reference solution

Build it yourself first; check after.

### Part A — Metrics with Prometheus (`obs-metrics`)
- Confirm the stack, then port-forward Prometheus as above. The three PromQL queries:
  ```promql
  sum by (namespace) (kube_pod_info)                              # pod count per namespace (spot sprawl)
  kube_pod_container_status_restarts_total > 0                    # restarting containers
  rate(apiserver_request_total{code=~"5.."}[5m])                  # API-server error rate
  ```
- **Status → Targets** shows the `ServiceMonitor`-selected endpoints; the scrape config lives in [`prometheus-values.yaml`](../lab-infra/observability/prometheus-values.yaml) as CRDs, never a hand-edited `prometheus.yml`.

### Part B — Logs with Loki (`obs-logs`)
- After deploying `demo-authlog.yaml`, in **Explore → Loki**:
  ```logql
  {namespace="oss500-apps", app="authlog"} |= "Failed password"
  ```
  the brute-force lines appear.
- Turn logs into a metric (the basis for a log alert):
  ```logql
  sum(rate({namespace="oss500-apps", app="authlog"} |= "Failed password" [1m]))
  ```
  the rate climbs — this is the shape you'll alert on in Part E (the deployed reference alert uses an equivalent metrics-counter form, `authlog_failed_logins_total`, exported by the demo app itself).

### Part C — Traces with OpenTelemetry + Tempo (`obs-traces`)
- Deploy `demo-traced-app.yaml`, generate traffic with the `curl` loop, then in Grafana **Explore → Tempo → Search** read the span tree (frontend → backend), stitched by `traceparent` propagation.
- The pipeline in [`otel-collector.yaml`](../lab-infra/observability/otel-collector.yaml): receivers (OTLP gRPC/HTTP) → processors (`memory_limiter`, `batch`) → exporters (`otlp/tempo`, `prometheus`, `otlphttp/loki`) — one Collector, three backends.

### Part D — Dashboards & correlation in Grafana (`obs-dashboards`)
- The provisioned **OSS-500 posture dashboard** (loaded via the sidecar labelled `grafana_dashboard` per `prometheus-values.yaml`'s `grafana.sidecar.dashboards`) has panels for API-server 5xx rate, pods-running-as-root count, denied/failed-auth log rate, and trace latency; drill down from the Loki panel to raw logs, and from a latency spike to the Tempo trace.
- `kubectl -n oss500-monitoring delete pod -l app.kubernetes.io/name=grafana` — dashboards (provisioned JSON) and all data (in Prometheus/Loki/Tempo) survive the pod restart, because Grafana holds no state of its own.

### Part E — Alerting rules + routing (`obs-alerting`)
- The complete `PrometheusRule`, in [`alert-rules.yaml`](../lab-infra/observability/alert-rules.yaml):
  ```yaml
  apiVersion: monitoring.coreos.com/v1
  kind: PrometheusRule
  metadata:
    name: oss500-security-rules
    namespace: oss500-monitoring
    labels:
      app.kubernetes.io/part-of: oss500
      # Selected by the kube-prometheus-stack ruleSelector (release label).
      release: oss500-kube-prometheus-stack
  spec:
    groups:
      - name: oss500.security
        rules:
          # obs-alerting: fire when the demo app's auth-failure rate (derived from Loki
          # logs via a recording rule / or from a metrics counter) crosses threshold for 2m.
          # `for: 2m` prevents flapping on a transient blip.
          - alert: AuthFailureSpike
            expr: sum(rate(authlog_failed_logins_total[2m])) > 0.2
            for: 2m
            labels:
              severity: critical
              team: security
            annotations:
              summary: "Spike in authentication failures"
              description: "Failed-login rate {{ $value | printf \"%.2f\" }}/s over 2m — possible brute force."
          # A restart/crash-loop can indicate a killed exploit or an unstable workload.
          - alert: KubePodCrashLooping
            expr: rate(kube_pod_container_status_restarts_total[15m]) > 0
            for: 15m
            labels:
              severity: warning
              team: security
            annotations:
              summary: "Pod {{ $labels.namespace }}/{{ $labels.pod }} is crash-looping"
  ```
- Trigger with `kubectl scale deploy/authlog -n oss500-apps --replicas=3`; watch `AuthFailureSpike` go `Pending`→`Firing` in Prometheus.
- Routing lives in [`alertmanager-config.yaml`](../lab-infra/observability/alertmanager-config.yaml): the alert's `team: security` / `severity: critical` labels match the `security-soc` receiver's route (grouped by `alertname`/`namespace`), which is exactly why those two labels belong on the rule.
- A silence for `alertname=AuthFailureSpike` in the Alertmanager UI mutes delivery without touching the underlying rule, which keeps firing in Prometheus — proof that rule (condition) and route (delivery) are independent layers.

If your `PrometheusRule` skipped `for:` entirely, it fires (and pages) on the first noisy scrape — the same "flapping" bug a missing hysteresis window causes anywhere else. If it skipped the `severity`/`team` labels, it fires correctly but silently falls through to the `default` receiver — a rule bug and a routing bug look identical from the dashboard, so always check both layers before declaring "no alert."

## What the exam asks
- Prometheus **pulls** metrics from `/metrics`; it doesn't receive pushes (Pushgateway is only for batch jobs). Metrics ≠ logs — high-cardinality data belongs in Loki.
- Loki indexes **labels, not content**; LogQL can derive metrics from logs. It's cheaper than Elastic but not a full-text engine — know Loki vs OpenSearch trade-offs.
- Traces answer **"where in the call graph"**; the OTel Collector is a deploy-once fan-out (OTLP in, Tempo/Prometheus/Loki out).
- Grafana **stores nothing** — it's visualization/correlation over the backends; dashboards live as JSON in Git.
- Alert **rule/condition (Prometheus) vs routing/delivery (Alertmanager)** are separate. `for:` prevents flapping; grouping/inhibition/silences are distinct routing behaviors. "Fires but no page" = routing bug, not a rule bug.
